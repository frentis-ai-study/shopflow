# ADR-0007: 이슈 → 브랜치 → PR 워크플로 강제

- **상태**: 수용(Accepted)
- **날짜**: 2026-07-16
- **관련**: 헌장 원칙 VIII, Spec Kit 단계 매핑

## 맥락 및 문제

변경 이력을 추적 가능하게 하고 리뷰 없는 무단 변경을 막을 협업 규율이 필요하다. 또한 이
규율을 Spec Kit 워크플로(사양→계획→태스크→구현)와 일관되게 연결해야 한다.

## 검토한 선택지

1. **main 직접 커밋 허용** — 빠르지만 이력 추적·리뷰 부재, 사고 위험.
2. **이슈 → 브랜치 → PR 강제 + 브랜치 보호** — 제안은 이슈, 작업은 브랜치, 완료는 PR.
3. **PR만 권장(강제 없음)** — 규율이 쉽게 무너짐.

## 결정

**모든 작업은 "제안은 이슈로, 작업은 브랜치로, 완료는 PR로" 흐름을 따른다.** main 브랜치
보호로 PR 병합을 강제하고, Spec Kit 단계와 다음과 같이 연결한다.

| Spec Kit 단계 | Git 워크플로 |
|---|---|
| `/speckit-specify` | GitHub 이슈(사양 제안) |
| `/speckit-plan`·`/speckit-tasks` | `feature/<이슈번호>-<슬러그>` 브랜치 |
| `/speckit-implement` | 브랜치 커밋(이슈 참조) |
| 완료 | PR 생성(`Closes #N`) → main 병합 |

## 근거

- 변경이 이슈·브랜치·PR로 분리되어 감사 가능해진다(원칙 VIII).
- 브랜치 보호가 규율을 사람 의지가 아닌 시스템으로 강제한다.
- 이슈 템플릿·PR 템플릿으로 헌장 준수 점검을 체크리스트화한다.

## 결과

- **좋은 점**: 추적성·리뷰 보장, Spec Kit과 1:1 대응, 무단 변경 차단.
- **나쁜 점/트레이드오프**: 단독 작업 시 절차 오버헤드가 있다. `main` 브랜치 보호는
  `enforce_admins: false`로 두어 비상 시 admin 우회를 허용한다(필요 시 강화 가능).
- 강제 장치: `.github/ISSUE_TEMPLATE/spec-proposal.md`, `.github/PULL_REQUEST_TEMPLATE.md`,
  main 브랜치 보호 규칙.
