# ADR-0008: 배송을 별도 바운디드 컨텍스트(Delivery)로 분리

- **상태**: 수용(Accepted)
- **날짜**: 2026-07-16
- **관련**: 헌장 원칙 V·VII, [ADR-0005](./0005-per-seller-order-splitting.md), spec FR-011·FR-022

## 맥락 및 문제

초기 모델에서 SubOrder(판매자별 하위주문)가 두 관심사를 겸했다: ① 판매자별 **정산·주문
단위**(금액), ② **배송 상태**(배송중/배송완료 전이·시각). 도메인을 바운디드 컨텍스트로 나눌
때, 정산과 물류(배송)를 한 엔티티가 함께 들고 있으면 컨텍스트 경계가 흐려진다.

## 검토한 선택지

1. **SubOrder에 배송 상태 유지** — 엔티티 하나로 단순하나, 정산·배송 관심사가 섞임. 향후
   송장·추적·부분배송을 넣으면 SubOrder가 비대해짐.
2. **배송을 별도 컨텍스트로 분리(Delivery 엔티티 신설)** — SubOrder는 정산 단위,
   Delivery는 배송 이행 단위(SubOrder와 1:1).

## 결정

**배송을 별도 바운디드 컨텍스트로 분리하고 `Delivery` 엔티티를 신설한다.** SubOrder는
판매자별 정산·주문 단위(sellerId, subtotalKrw)만 책임지고, 배송 상태·시각은 Delivery가
담당한다.

- Delivery: `subOrderId`(1:1), `status`(`PENDING→SHIPPING→DELIVERED`), `shippedAt`,
  `deliveredAt`.
- 결제 완료 시 각 SubOrder에 대해 `PENDING` Delivery를 생성한다.
- 판매자의 배송 전이(FR-011)는 Delivery 상태 전이로 처리한다.
- 주문(Order) 전체 상태는 연결된 Delivery 상태들의 집계로 산출한다.

## 근거

- 정산(SubOrder)과 물류(Delivery)의 관심사가 명확히 분리된다(원칙 V).
- 배송·정산·환불을 판매자 단위로 관리하는 원칙 VII와 정합하며, 향후 송장·추적번호·부분배송을
  배송 컨텍스트 안에서 확장할 수 있다.
- 코드도 `delivery` 패키지로 독립해 팀·모듈 경계가 분명해진다.

## 결과

- **좋은 점**: 컨텍스트 경계 명확, 배송 도메인 확장 용이, SubOrder 단순화.
- **나쁜 점/트레이드오프**: 엔티티·테이블이 하나 늘고 SubOrder↔Delivery(1:1) 조회가 추가된다.
  MVP에서는 배송 상태 전이만 쓰지만, 컨텍스트 분리의 이점이 이 비용을 상회한다고 판단한다.
- ADR-0005(판매자별 분리)는 유효하며, 본 ADR은 그 하위주문에서 배송 관심사를 분리하는
  후속 결정이다.
