# ADR-0009: 판매자 프로필과 사업자 유형 모델링

- **상태**: 수용(Accepted)
- **날짜**: 2026-07-16
- **관련**: 헌장 원칙 VII, spec FR-006·FR-009, [ADR-0005](./0005-per-seller-order-splitting.md)

## 맥락 및 문제

초기 모델은 판매자를 `Product.sellerId → User(SELLER 역할)`로만 표현해, 판매자가 **개인
사업자인지 법인인지** 구분하거나 사업자등록번호·상호·대표자 같은 정산·세무에 필요한 정보를
담을 수 없었다. 마켓플레이스에서 이는 정산·세금계산서·입점 관리의 기반이 빠진 상태다.

## 검토한 선택지

1. **User에 판매자 필드 추가** — 단순하나 계정과 판매자 관심사가 섞이고 유형별 필수값을
   표현하기 어렵다.
2. **Seller(판매자 프로필) 엔티티 신설(User와 1:1)** — 판매 활동 정보를 분리, 유형별 검증 가능.
3. **사업자 여부 boolean만** — 최소 정보. 법인/개인사업자 구분·사업자등록번호 관리를 못함.

## 결정

**계정 컨텍스트에 `Seller`(판매자 프로필)를 신설하고 User와 1:1로 연결한다.** 핵심 필드:

- `sellerType`: `INDIVIDUAL`(개인) / `SOLE_PROPRIETOR`(개인사업자) / `CORPORATION`(법인).
- `storeName`(스토어 표시명), `contactPhone`, `contactEmail`, `status`(ACTIVE/SUSPENDED).
- `businessRegistrationNumber`(사업자등록번호), `representativeName`(대표자) —
  **`sellerType != INDIVIDUAL`이면 필수**, 형식·중복 검증.

`Product.sellerId`·`SubOrder.sellerId`는 Seller를 참조한다. 정산 계좌·세금계산서 정보는
**정산 기능(후속 이슈)** 에서 확장한다.

## 근거

- 개인사업자/법인 구분과 사업자등록번호를 정식으로 담아 정산·세무의 기반을 마련한다(원칙 VII).
- 계정(신원)과 판매자(사업 활동) 관심사를 분리한다.
- 유형별 필수값 검증을 도메인에서 강제할 수 있다.

## 결과

- **좋은 점**: 판매자 유형별 처리·정산 확장 용이, 입점 상태(SUSPENDED) 관리 가능.
- **나쁜 점/트레이드오프**: 엔티티·가입/입점 폼이 늘어난다. 사업자등록번호 진위 확인(국세청
  연동)은 이번 범위 밖이며 형식 검증까지만 한다.
