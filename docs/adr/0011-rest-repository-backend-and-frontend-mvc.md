# ADR-0011: REST 리포지토리 백엔드 + 프론트 제어 MVC 레이어 분리

- **상태**: 수용(Accepted) — [ADR-0001](./0001-server-side-rendering-spring-thymeleaf.md)의 전달 계층을 대체·보완, [ADR-0010](./0010-extraction-ready-modular-monolith.md)의 프론트 분리와 정합
- **날짜**: 2026-07-16
- **관련**: 헌장 원칙 II·V, C4 컨테이너/컴포넌트 다이어그램

## 맥락 및 문제

초기 설계(ADR-0001)는 전통적 MVC(Controller→Service→Repository)로 Thymeleaf를 직접 렌더했다.
그러나 목표 아키텍처(ADR-0010)는 프론트 분리를 지향하고, 도메인 데이터 접근을 표준 REST로
노출하면 프론트·외부 연동·향후 서비스 분리가 쉬워진다. 전달 계층을 재설계할 필요가 있다.

## 검토한 선택지

1. **전통적 MVC 유지** — 뷰와 도메인 접근이 컨트롤러에 결합, 프론트 분리·API화가 어렵다.
2. **Spring Data REST(RepositoryRestResource) 기반 백엔드 + 프론트 제어 MVC 분리** — 리포지토리
   를 REST 리소스로 노출하고, 화면 제어는 별도 MVC 레이어(BFF)가 REST를 소비.
3. **완전 수기 REST 컨트롤러** — 유연하나 CRUD 보일러플레이트 증가.

## 결정

**두 계층으로 나눈다.**

- **백엔드 REST API**: 애그리거트 CRUD는 **Spring Data REST**(`@RepositoryRestResource`)로 REST
  리소스를 자동 노출한다. 도메인 연산(체크아웃, 배송 상태 전이, 판매 상태 전환, 재고 선점
  등 CRUD가 아닌 것)은 **커스텀 REST 컨트롤러 + 애플리케이션 서비스**로 제공한다. 컨텍스트별
  모듈 구조·경계는 유지(ADR-0005/0008/0010).
- **프론트 제어 MVC 레이어**: Spring MVC(+Thymeleaf) 레이어가 화면 렌더·폼·세션·플로우를
  담당하고, 백엔드 REST API를 호출(BFF)한다. 지금은 같은 배포 단위 내 별도 모듈, 추후 별도
  인스턴스(프론트 컨테이너)로 분리(ADR-0010).

## 근거

- 리포지토리 REST 노출로 CRUD 보일러플레이트를 줄이고 표준화된 리소스 계약을 얻는다.
- 화면 제어(MVC)와 도메인 API(REST)를 분리해 프론트 분리·외부 연동·서비스 추출이 쉬워진다.
- 도메인 규칙(멱등·선점·상태 전이·권한)은 커스텀 컨트롤러+서비스에 남겨 무결성을 지킨다(원칙 II·III).

## 결과

- **좋은 점**: 표준 REST 리소스, 프론트/백 경계 명확, 목표 아키텍처(ADR-0010)로 자연 확장.
- **나쁜 점/트레이드오프**:
  - Spring Data REST의 자동 노출 범위를 신중히 제한해야 한다(민감 필드·쓰기 권한). 노출
    정책·프로젝션·검증을 명시하고, 상태 변경 무결성이 필요한 리소스는 커스텀 컨트롤러로만
    변경한다.
  - 프론트 제어 MVC ↔ 백엔드 REST 간 호출 계약·인증(세션/토큰) 설계가 추가된다.
  - ADR-0001의 "Thymeleaf SSR"은 프론트 제어 MVC 레이어에서 유지되되, 도메인 접근 방식은
    본 ADR로 대체된다.
