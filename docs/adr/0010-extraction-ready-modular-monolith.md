# ADR-0010: 모듈러 모놀리스 확정 (백엔드 인스턴스 분리 Non-goal, 프론트만 별도 인스턴스)

- **상태**: 수용(Accepted) — 2026-07-16 개정(v2). 이전 버전은 "컨텍스트별 별도 인스턴스"를
  목표로 두었으나, 네거티브 검토([reviews/2026-07-16-negative-review.md](../../specs/001-marketplace-core/reviews/2026-07-16-negative-review.md))
  결과 이를 **Non-goal로 강등**한다.
- **날짜**: 2026-07-16 (개정)
- **관련**: 헌장 원칙 V, [ADR-0011](./0011-rest-repository-backend-and-frontend-mvc.md), 바운디드 컨텍스트([ADR-0005](./0005-per-seller-order-splitting.md), [ADR-0008](./0008-delivery-bounded-context.md))

## 맥락 및 문제

초기(v1)에는 "바운디드 컨텍스트별 별도 배포 인스턴스"를 목표로 두었다. 그러나 검토 결과,
코어 흐름(재고 선점의 원자적 조건부 UPDATE, 체크아웃 단일 트랜잭션, 주문 상태 집계)이 모두
**단일 DB 원자성에 의존**해, 컨텍스트를 인스턴스로 쪼개는 순간 사가·보상·아웃박스로 **코어를
재작성**해야 함이 드러났다. "선점·멱등이 분리를 대비한다"는 것은 재시도 안전성일 뿐 분리
원자성을 대비하지 못한다. 인스턴스 분리를 목표로 유지하면 원칙 V(단순성)와 정면충돌한다.

또한 화면 디자인은 **외부에서 별도로 진행**되므로, 백엔드 서비스를 먼저 완성하고 디자인이
완료되는 시점에 프론트엔드를 붙이는 순서가 현실적이다.

## 결정

**백엔드는 단일 배포의 모듈러 모놀리스로 확정한다. 바운디드 컨텍스트별 별도 인스턴스 분리는
Non-goal(비목표)로 한다.** 다만 **프론트엔드는 별도 인스턴스**로 두어, 외부 디자인 완료 후
백엔드 REST API를 소비하도록 붙인다(ADR-0011의 프론트 제어 레이어를 별도 배포로).

준수 규칙(개정):
- **모듈 경계 = 컨텍스트 경계**: 패키지(account, product, order, payment, inventory,
  delivery, common)를 컨텍스트로 유지. 다른 컨텍스트의 내부 클래스 직접 의존은 지양하고
  애플리케이션 서비스 인터페이스를 경유한다(가독성·응집).
- **단일 DB, 단일 트랜잭션 허용**: 컨텍스트 간 **조인·트랜잭션을 허용**한다(인스턴스 분리를
  전제하지 않으므로). 코어 흐름의 원자성을 단일 DB 트랜잭션으로 보장한다.
- **컨텍스트 내부 무결성 우선**: 재고(products.reserved 조건부 UPDATE), 주문 상태 집계 등은
  단일 DB에서 정합적으로 처리한다. 크로스 컨텍스트 쓰기는 서비스 인터페이스로 감싼다.
- **프론트/백 분리는 유지**: 백엔드는 REST API를 노출(ADR-0011), 프론트엔드는 별도 인스턴스
  (외부 디자인 완료 후 구축). 개발 순서는 **백엔드 서비스 우선 → 프론트엔드 후행**.

## 근거

- 코어 원자성을 단일 DB로 보장해 초과판매·이중결제 무결성을 단순·확실하게 지킨다(원칙 III·V).
- 인스턴스 분리(사가·이벤트·아웃박스)의 큰 복잡도를 도입하지 않는다 — 실제 필요가 입증되면
  그때 별도 ADR로 다룬다(YAGNI).
- 프론트를 별도 인스턴스로 두면 외부 디자인 트랙과 병렬 진행이 가능하고, 백엔드는 REST 계약만
  안정화하면 된다.

## 결과

- **좋은 점**: 코어 무결성 단순·확실, 운영 복잡도 최소, 외부 디자인과 병렬 개발.
- **트레이드오프/유의**:
  - 훗날 특정 컨텍스트만 스케일아웃해야 하면 그때 분리 비용을 지불한다(현재는 불필요).
  - 프론트/백 분리는 유지되므로 **백엔드 REST 계약**(엔드포인트·페이로드·인증 전파)을 명확히
    문서화해야 한다(현재 공백 — 후속 보강 대상).
  - ADR-0001(SSR)·ADR-0011(REST+프론트 제어)은 유효하되, "백엔드 인스턴스 분리"만 폐기한다.
- **파생 결정**: 개발 순서를 백엔드 서비스 우선으로 재편(tasks.md 재구성), 프론트엔드는 디자인
  완료 시점에 별도 인스턴스로 구축한다.
