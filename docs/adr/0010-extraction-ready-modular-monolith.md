# ADR-0010: 추출 가능한 모듈러 모놀리스 (목표: 컨텍스트별 별도 인스턴스)

- **상태**: 수용(Accepted) — [ADR-0001](./0001-server-side-rendering-spring-thymeleaf.md)의 배포·경계 측면을 보완
- **날짜**: 2026-07-16
- **관련**: 헌장 원칙 V, 바운디드 컨텍스트 구성([ADR-0005](./0005-per-seller-order-splitting.md), [ADR-0008](./0008-delivery-bounded-context.md))

## 맥락 및 문제

목표 아키텍처로 **바운디드 컨텍스트별 별도 배포 인스턴스**(계정/상품/주문/결제/재고/배송)와
**추후 프론트엔드 분리**를 가정한다. 그러나 MVP부터 마이크로서비스로 가면 인프라·운영·분산
트랜잭션 복잡도가 급증해 원칙 V(단순성)와 충돌한다. 목표를 지키면서 초기 비용을 통제할
방법이 필요하다.

## 검토한 선택지

1. **처음부터 서비스별 별도 인스턴스(마이크로서비스 now)** — 목표엔 부합하나 MVP에 과한
   운영·분산 복잡도.
2. **추출 가능한 모듈러 모놀리스** — 한 배포 단위로 시작하되, 컨텍스트별 스키마·모듈 경계·
   ID 참조·명시적 내부 인터페이스로 짜서 나중에 인스턴스로 **분리(추출)** 가능하게 설계.
3. **경계 없는 모놀리스** — 가장 단순하나 목표(분리)와 배치되어 나중에 큰 재작업.

## 결정

**"추출 가능한 모듈러 모놀리스"로 구현하고, 목표 아키텍처(컨텍스트별 별도 인스턴스 + 프론트
분리)를 설계 원칙으로 명시한다.** 준수 규칙:

- **모듈 경계 = 컨텍스트 경계**: 패키지(account, product, order, payment, inventory,
  delivery, common)를 컨텍스트로 고정. 다른 컨텍스트의 내부 클래스에 직접 의존 금지.
- **데이터 소유 분리**: 컨텍스트별 스키마(또는 스키마 네임스페이스). **컨텍스트 간 조인·하드
  FK 금지**, 참조는 ID로만(느슨한 결합).
- **명시적 인터페이스**: 컨텍스트 간 상호작용은 애플리케이션 서비스 인터페이스/도메인 이벤트로.
  in-process 호출이지만 원격 호출로 바꿔도 계약이 유지되게 설계.
- **프론트 분리 대비**: 도메인/애플리케이션 로직을 뷰에서 분리(원칙 V). 현재는 Thymeleaf SSR
  포함(ADR-0001), 추후 프론트를 별도 인스턴스로 떼면 백엔드는 API를 노출.

## 근거

- 목표(컨텍스트별 분리)를 설계에 못박되, MVP 운영 비용을 폭증시키지 않는다(원칙 V).
- 현재 설계가 이미 forward-compatible: 컨텍스트 간 **ID 참조**, 재고 **선점(TTL)** 과 결제
  **멱등키** 는 분산 환경의 예약·재시도를 전제한다.
- 분리 시 변경 범위를 모듈 경계로 국소화한다.

## 결과

- **좋은 점**: 단일 배포의 단순함 + 미래 분리 대비. 경계 위반을 리뷰/모듈 규칙으로 차단.
- **나쁜 점/트레이드오프**:
  - 컨텍스트 간 조인 금지로 일부 조회가 여러 번의 조회·조합이 된다.
  - **체크아웃의 분산 전이 리스크**: 지금은 재고·주문·결제·배송이 한 DB 트랜잭션이지만,
    인스턴스로 분리되면 단일 ACID가 불가해 **사가(saga) + 보상 트랜잭션**이 필요하다. 선점·
    멱등 설계로 대비되어 있으나, 실제 분리 시 오케스트레이션 ADR을 추가한다(후속).
  - ADR-0001의 "단일 스택·모놀리스"는 유효하되, 프론트 분리·인스턴스 분리 측면은 본 ADR이
    보완한다.
