# ADR-0001: 서버사이드 렌더링(Spring Boot + Thymeleaf) 채택

- **상태**: 수용(Accepted) — 배포·프론트 분리는 [ADR-0010](./0010-extraction-ready-modular-monolith.md), 전달 계층(REST 백엔드 + 프론트 제어 MVC)은 [ADR-0011](./0011-rest-repository-backend-and-frontend-mvc.md)이 보완/대체
- **날짜**: 2026-07-16
- **관련**: 헌장 원칙 V(단순성·계층형 아키텍처), 기술·보안 제약

## 맥락 및 문제

ShopFlow의 화면 전달 방식을 정해야 한다. 초기 팀 역량과 MVP 속도, 그리고 결제·재고 등
서버 중심 로직의 비중을 고려해 프론트엔드 아키텍처를 결정한다.

## 검토한 선택지

1. **Spring Boot + Thymeleaf (서버사이드 렌더링)** — 서버가 HTML을 렌더. 백엔드 단일 스택.
2. **SPA 분리(React + REST/GraphQL API)** — 프론트/백 분리, 클라이언트 렌더.
3. 하이브리드(부분 SSR + 아일랜드).

## 결정

**Spring Boot + Thymeleaf 기반 서버사이드 렌더링을 채택한다.** 백엔드는 Spring MVC /
Spring Data / Spring Security로 구성하고, 화면은 로직을 최소화한 Thymeleaf 템플릿으로
렌더한다.

## 근거

- 결제·재고·주문 등 핵심 가치가 서버 로직에 몰려 있어, 클라이언트 렌더 분리의 이득이 작다.
- 단일 스택으로 계층 분리(컨트롤러→서비스→리포지토리)를 단순하게 유지할 수 있다(원칙 V).
- 인증·CSRF·서버측 검증을 Spring Security로 일관되게 강제하기 쉽다(원칙 II).
- MVP 개발·배포가 빠르다.

## 결과

- **좋은 점**: 낮은 복잡도, 빠른 MVP, 보안 기본기 강제 용이, 뷰-도메인 경계 명확.
- **나쁜 점/트레이드오프**: 리치한 클라이언트 상호작용에는 제약. 향후 고도화 시 부분적으로
  클라이언트 스크립트/아일랜드 도입이 필요할 수 있다.
- 뷰 템플릿에 비즈니스 로직을 두지 않는다는 제약을 코드리뷰에서 강제한다(원칙 V).
