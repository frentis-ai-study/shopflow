# 퀵스타트: 멀티셀러 핵심 커머스 검증 가이드

**기능**: [spec.md](./spec.md) · **계획**: [plan.md](./plan.md) · **관련 이슈**: #2

이 문서는 기능이 엔드투엔드로 동작함을 증명하는 실행·검증 시나리오다. 구현 코드는 담지 않고,
실행 방법과 기대 결과만 기술한다(상세는 [data-model.md](./data-model.md),
[contracts/](./contracts/) 참조).

## 사전 준비

- Java 21, Maven (프로젝트에 `mvnw` 래퍼 포함)
- PostgreSQL 16 (로컬 또는 Testcontainers)
- dev-ports 규칙: 포트는 환경변수로 주입. 최초 1회 셋업:
  ```bash
  dev-init                 # .envrc(use_dev_ports) 생성 + direnv allow
  direnv exec . env | grep -E 'BACKEND_PORT|POSTGRES_PORT'
  ```
  기본 슬롯 0 → `BACKEND_PORT=18000`, `POSTGRES_PORT=15432`.

## 설정

- `application.yml`은 다음을 환경변수에서 읽는다(하드코딩 금지):
  - 서버 포트 ← `BACKEND_PORT`(기본 18000)
  - DB 접속 ← `POSTGRES_PORT`(기본 15432), DB 이름/자격증명은 시크릿/환경변수
- 스키마는 Flyway 마이그레이션(`db/migration`)으로 자동 적용.

## 실행

```bash
# DB 준비(예: 로컬 도커) 후
direnv exec . ./mvnw spring-boot:run        # http://localhost:${BACKEND_PORT:-18000}
```

## 검증 시나리오

각 시나리오는 사양의 수락 기준·성공 기준과 대응한다.

### S1. 판매자 상품 등록 → 노출 (US2 / SC-002)

1. 판매자로 가입·로그인 → `/seller/products/new`에서 상품(재고 3) 등록.
2. **기대**: 상품이 `ON_SALE`로 저장되고 `/products` 목록에 노출.

### S2. 구매자 결제 해피패스 (US1 / SC-001, SC-003)

1. 구매자로 로그인 → 상품을 장바구니에 담고 `/checkout`.
2. 배송지 입력 후 결제(모의 게이트웨이 승인 조건).
3. **기대**: 주문이 `PAID`로 생성, 재고 3→2로 확정 차감, 완료 페이지에 주문번호 표시.

### S3. 재고 부족·초과 판매 방지 (엣지 / SC-003)

1. 재고 1 상품을 두 세션이 동시에 결제 시도.
2. **기대**: 한 세션만 성공, 다른 세션은 품절 안내. 최종 재고는 음수가 되지 않음(초과 판매 0).

### S4. 멱등 결제 (엣지 / SC-004)

1. 동일 `idempotencyKey`로 `/checkout` POST를 2회(재시도 시뮬레이션).
2. **기대**: 주문은 1건만 생성, 두 번째 요청은 기존 주문 결과로 응답(이중 결제 없음).

### S5. 결제 미완료 TTL 복원 (엣지 / SC-005)

1. 결제 직전 재고 선점만 발생시키고 결제를 완료하지 않음.
2. 10분(TTL) 경과(또는 테스트에서 만료 시각 앞당김) 후 스윕 실행.
3. **기대**: 선점이 `RELEASED`로 바뀌고 가용 재고가 원복.

### S6. 결제 실패 시 재고 보존 (UC-09 대안)

1. 모의 게이트웨이가 거절하도록 트리거해 결제.
2. **기대**: 주문 미생성, 선점 해제로 재고 원복, 실패 사유 안내.

### S7. 판매자별 이행 분리·배송 전이 (US4 / SC-006)

1. 서로 다른 두 판매자 상품을 한 번에 결제.
2. `/orders/{id}`에서 판매자별 이행 단위 2개 확인.
3. 각 판매자가 `/seller/orders`에서 본인 건만 보고 `ship`→`deliver` 전이.
4. **기대**: 구매자 주문 상세에 판매자별 상태가 각각 반영, 배송완료 시각 기록.

### S8. 권한 격리 (SC-007)

1. 판매자 A가 판매자 B의 상품 수정/주문 접근 시도.
2. 비로그인으로 `/checkout`·`/seller/products` 접근 시도.
3. **기대**: 각각 403 / `/login` 리다이렉트로 차단.

## 자동화 테스트 대응 (원칙 I)

- 단위: 가격 합계·재고 선점 판정·상태 전이 규칙·멱등 결과 동일성(순수 도메인).
- 통합(Testcontainers): S2~S8의 흐름을 결정적으로 재현.
- 웹 슬라이스: 인증/CSRF/권한(S8).
