# Phase 0 리서치: 멀티셀러 핵심 커머스

**기능**: [spec.md](./spec.md) · **계획**: [plan.md](./plan.md) · **관련 이슈**: #2

기술 컨텍스트의 미해결 항목을 결정으로 확정한다. 각 항목은 결정·근거·검토한 대안 형식.
확정 결과 중 아키텍처적으로 유의미한 것은 ADR로도 승격한다.

---

## R1. 재고 선점의 동시성 제어 방식

- **결정**: **원자적 조건부 UPDATE**를 1차 방어선으로 사용한다. 상품 재고 행에 대해
  `UPDATE product SET reserved = reserved + :qty WHERE id = :id AND (stock - reserved) >= :qty`
  형태의 단일 조건부 갱신으로 선점하고, 영향 행 수가 0이면 재고 부족으로 처리한다. 선점·확정·
  해제는 하나의 트랜잭션 안에서 수행한다.
- **근거**: 단일 조건부 UPDATE는 행 수준 잠금이 암묵 적용되어 초과 판매를 원자적으로 막고,
  별도 락 인프라가 필요 없다. 결과가 결정적이라 테스트가 쉽다(SC-003).
- **검토한 대안**:
  - 비관적 락(`SELECT ... FOR UPDATE` 후 검증·갱신): 명시적이나 왕복이 늘고 핫 로우에서
    대기 증가. 조건부 UPDATE로 충분.
  - 낙관적 락(version + 재시도): 경합이 잦은 한정 재고에서 재시도 폭증 위험.
- **ADR**: [ADR-0004](../../docs/adr/0004-stock-reservation-ttl.md)의 후속 세부 결정으로 기록.

## R2. 재고 선점 만료(TTL) 처리

- **결정**: **스케줄 스윕 + 지연 판정 병행.** (1) `@Scheduled` 배치(예: 60초 주기)가 만료된
  선점을 찾아 `reserved`를 감소시켜 재고를 복원한다. (2) 조회·선점 시점에도 만료된 선점은
  무효로 간주(지연 판정)해 스윕 지연 창을 보완한다.
- **근거**: 스케줄 스윕은 미결제 이탈 재고를 확실히 100% 복원한다(SC-005). 지연 판정을 더해
  스윕 주기 사이의 오차를 없앤다.
- **검토한 대안**:
  - 지연 판정만: 만료 선점이 조회되기 전까지 `reserved`가 부풀려져 가용 재고 계산이 왜곡.
  - 외부 TTL 저장소(Redis 등): 인프라 추가. MVP엔 과함. 후속 확장 여지로 둠.

## R3. 멱등키 발급·저장·조회

- **결정**: **서버 발급 멱등키 + 전용 테이블.** 체크아웃 진입 시 서버가 멱등키(UUID)를 발급해
  결제 폼의 숨은 필드로 전달한다. 결제 처리 시 `payment_idempotency(key UNIQUE, request_hash,
  status, order_id, created_at)`에 먼저 기록을 시도한다. 유니크 제약 충돌(동일 키 재요청)이면
  기존 결과(주문 id·상태)를 그대로 반환한다.
- **근거**: 서버 발급으로 키 예측·위조를 줄이고, DB 유니크 제약이 "정확히 한 번"을 저장소
  수준에서 강제한다(FR-018, SC-004). 별도 락 불필요([ADR-0003](../../docs/adr/0003-idempotent-payment-exactly-once.md)).
- **검토한 대안**:
  - 클라이언트 발급 키: 단순하나 신뢰·중복 위험. 서버 발급이 안전.
  - 분산 락: 인프라 복잡, 저장소 유니크 제약으로 충분.
- **보존**: 멱등 레코드는 최소 주문 조회 기간 동안 보존(구체 기간은 운영 정책, 기본 90일 가정).

## R4. 주문·배송 상태 기계 (배송 컨텍스트 분리)

- **결정**: **책임 분리 + 2계층 집계.**
  - 배송(Delivery) 상태: `PENDING`(배송대기) → `SHIPPING`(배송중) → `DELIVERED`(배송완료).
    결제 완료 시 SubOrder마다 `PENDING` Delivery를 생성한다. 허용 전이만 강제(건너뛰기·역행
    거절).
  - 하위주문(SubOrder)은 상태를 갖지 않고 **판매자별 정산·주문 단위**만 책임진다.
  - 주문(Order) 전체 상태: 하위주문에 연결된 Delivery 상태들의 집계(예: 전부 DELIVERED면
    주문도 완료 표시).
  - 각 전이는 감사 로그(전이 주체·시각·이전/이후 상태)를 남긴다(FR-022, 원칙 IV).
- **근거**: 배송을 별도 바운디드 컨텍스트로 분리하면 정산(SubOrder)과 물류(Delivery)의 관심사가
  갈리고, 향후 송장·추적·부분배송을 배송 컨텍스트 안에서 확장할 수 있다([ADR-0005](../../docs/adr/0005-per-seller-order-splitting.md),
  [ADR-0008](../../docs/adr/0008-delivery-bounded-context.md)).
- **검토한 대안**: SubOrder에 배송 상태를 함께 두기 → 한 엔티티가 정산·배송 두 관심사를 겸해
  컨텍스트 경계가 흐려짐.

## R5. 결제 어댑터 계약

- **결정**: **`PaymentGateway` 인터페이스** 한 개로 추상화하고 MVP는 `MockPaymentGateway`
  구현을 사용한다. 계약(개념): `authorize(멱등키, 금액, 주문참조) → 결과(승인/거절, 게이트웨이
  참조)`. 모의 구현은 규칙(예: 특정 금액·플래그)으로 성공/실패를 결정적으로 시뮬레이션한다.
- **근거**: 실 PSP 교체 시 도메인 변경 없음([ADR-0002](../../docs/adr/0002-payment-external-delegation-mock-gateway.md)).
  멱등키를 계약에 포함해 R3와 정합.
- **검토한 대안**: 결제 로직을 서비스에 직접 삽입 → 실 PSP 도입 시 침습적. 어댑터가 우수.

## R6. 인증·세션·비밀번호

- **결정**: **Spring Security 폼 로그인 + 세션 기반 인증**, 비밀번호는 **BCrypt** 해시.
  상태 변경 엔드포인트에 CSRF 토큰·인가 강제. 역할은 단일 계정에 구매자/판매자 복수 보유(FR-004).
- **근거**: 서버사이드 렌더링(ADR-0001)에 세션 인증이 자연스럽고 Spring Security로 CSRF·인가를
  일관 강제(원칙 II).
- **검토한 대안**: JWT/무상태 → SPA·API 분리에 적합하나 본 구조엔 불필요한 복잡성.

## R7. 금액 표현 타입

- **결정**: **정수 원(KRW) — `long`(또는 도메인 값객체 `Money`)** 로 저장·계산. 원은 최소
  단위가 1원이라 소수부가 없다.
- **근거**: 이진 부동소수점 배제([ADR-0006](../../docs/adr/0006-money-fixed-precision.md)),
  단일 통화 원 기준(spec 가정). 값객체로 감싸 연산 안전성 확보.
- **검토한 대안**: `BigDecimal` → 다중 통화·소수 통화까지 대비하나 현재 범위엔 과함. 후속
  다중 통화 도입 시 재검토.

## R8. 빌드·마이그레이션·설정

- **결정**: **Maven** 빌드, **Flyway** 스키마 마이그레이션, `application.yml`에서 포트·DB·
  시크릿을 환경변수(`BACKEND_PORT`, `POSTGRES_PORT`, DB 자격증명)로 바인딩. Maven Wrapper
  (`mvnw`)를 포함한다.
- **근거**: 버전 관리되는 마이그레이션(원칙 III·기술 제약), 포트·시크릿 외부화(dev-ports 규칙).
  Maven은 Spring Boot 표준 구성이 잘 알려져 있고 `pom.xml` 선언이 명시적이다.
- **검토한 대안**: Gradle도 가능하나 사용자 요청에 따라 Maven 채택. 수기 DDL 대신 Flyway로 이력화.

## R9. 아키텍처 경계 (추출 가능한 모듈러 모놀리스)

- **결정**: 컨텍스트별 패키지·스키마·ID 참조·명시적 인터페이스로 짠 **추출 가능한 모듈러
  모놀리스**로 구현한다. 목표는 컨텍스트별 별도 인스턴스 + 프론트 분리([ADR-0010](../../docs/adr/0010-extraction-ready-modular-monolith.md)).
- **근거**: 목표(분리)를 설계에 반영하되 MVP 운영 비용을 통제. 컨텍스트 간 조인·하드 FK
  금지, 상호작용은 애플리케이션 서비스/이벤트로.
- **체크아웃 일관성**: 현재는 재고·주문·결제·배송이 한 DB 트랜잭션. 인스턴스 분리 시 단일
  ACID가 불가 → 사가 + 보상 트랜잭션 필요. 선점(TTL)·멱등(R2/R3) 설계가 이를 대비한다. 실제
  분리 시 오케스트레이션 ADR을 추가한다(후속).
