# 유즈케이스 명세서: 멀티셀러 핵심 커머스 (구매·판매)

**기능**: [spec.md](./spec.md) · **다이어그램**: [diagrams/usecase-marketplace-core.png](./diagrams/usecase-marketplace-core.png)

**관련 이슈**: #2 · **작성일**: 2026-07-16 · **상태**: Draft

이 문서는 유즈케이스 다이어그램의 각 유즈케이스를 상세히 서술한다. 각 항목은 사양서의 기능
요구사항(FR)·성공 기준(SC)과 연결된다.

---

## 액터 (Actors)

| 액터 | 설명 |
|---|---|
| **방문자 (Guest)** | 아직 로그인하지 않은 사용자. 가입·로그인·상품 탐색/상세 보기가 가능하다. |
| **구매자 (Buyer)** | 로그인한 사용자가 구매 역할로 행동. 장바구니·결제·내 주문 조회. 방문자를 상속. |
| **판매자 (Seller)** | 로그인한 사용자가 판매 역할로 행동. 상품 등록·수정·판매 상태 전환·자기 주문 이행. 방문자를 상속. |
| **모의 결제 게이트웨이** (외부) | 결제 성공/실패/멱등을 시뮬레이션하는 외부 시스템. 실 PSP 연동은 후속 이슈. |

> 하나의 계정은 구매자·판매자 역할을 동시에 가질 수 있다(FR-004).

## UC-01 회원가입

- **주요 액터**: 방문자
- **목적**: 방문자가 계정을 만들어 구매·판매를 시작할 수 있게 한다.
- **관련 요구사항**: FR-001, FR-002
- **사전조건**: 로그인하지 않은 상태.
- **사후조건(성공)**: 새 계정이 생성되고 로그인할 수 있다.

**기본 흐름**
1. 방문자가 가입 화면에서 이메일·비밀번호를 입력한다.
2. 시스템이 이메일 형식과 중복 여부를 검증한다.
3. 시스템이 계정을 생성하고 가입 완료를 안내한다.

**대안/예외 흐름**
- 2a. 이메일 형식이 잘못됨 → 오류 메시지를 표시하고 재입력을 요청한다.
- 2b. 이미 가입된 이메일 → "이미 사용 중인 이메일" 안내 후 로그인/재입력을 유도한다.
- 2c. 비밀번호가 정책(길이 등)에 미달 → 기준을 안내하고 재입력을 요청한다.

## UC-02 로그인 / 로그아웃

- **주요 액터**: 방문자(로그인), 구매자·판매자(로그아웃)
- **목적**: 인증 세션을 시작·종료한다.
- **관련 요구사항**: FR-003, FR-005
- **사전조건**: (로그인) 가입된 계정이 있음.
- **사후조건(성공)**: (로그인) 인증 세션 시작, 보호된 기능 접근 가능. (로그아웃) 세션 종료.

**기본 흐름 (로그인)**
1. 사용자가 이메일·비밀번호를 입력한다.
2. 시스템이 자격증명을 검증한다.
3. 시스템이 인증 세션을 시작하고 홈/직전 화면으로 이동시킨다.

**대안/예외 흐름**
- 2a. 자격증명 불일치 → "이메일 또는 비밀번호가 올바르지 않습니다" 안내(어느 쪽이 틀렸는지
  특정하지 않음, 보안).
- 로그아웃: 사용자가 로그아웃을 요청하면 세션을 종료하고 공개 화면으로 전환한다.

## UC-03 상품 탐색·검색

- **주요 액터**: 방문자(구매자·판매자 포함)
- **목적**: 판매중 상품을 찾아본다.
- **관련 요구사항**: FR-012, FR-008(판매중지 상품 미노출)
- **사전조건**: 없음(비로그인 허용).
- **사후조건(성공)**: 조건에 맞는 판매중 상품 목록이 표시된다.

**기본 흐름**
1. 사용자가 상품 목록 화면을 연다.
2. 시스템이 판매중 상태의 상품만 목록으로 보여준다.
3. 사용자가 이름으로 검색한다.
4. 시스템이 검색어에 맞는 판매중 상품을 보여준다.

**대안/예외 흐름**
- 2a. 판매중 상품이 없음 → 빈 목록 안내를 표시한다.
- 4a. 검색 결과 없음 → "검색 결과가 없습니다" 안내를 표시한다.

## UC-04 상품 상세 보기

- **주요 액터**: 방문자
- **목적**: 상품의 상세 정보를 확인한다.
- **관련 요구사항**: FR-013
- **사전조건**: 대상 상품이 존재.
- **사후조건(성공)**: 상품의 가격·설명·재고 유무·판매자가 표시된다.

**기본 흐름**
1. 사용자가 목록/검색에서 상품을 선택한다.
2. 시스템이 상품 상세(가격, 설명, 재고 유무, 판매자)를 보여준다.

**대안/예외 흐름**
- 2a. 상품이 판매중지/삭제됨 → 접근 불가 또는 "판매하지 않는 상품" 안내.

## UC-05 상품 등록

- **주요 액터**: 판매자
- **목적**: 판매할 상품을 등록해 구매자에게 노출한다.
- **관련 요구사항**: FR-005, FR-006
- **사전조건**: 판매자 역할로 로그인.
- **사후조건(성공)**: 상품이 "판매중"으로 저장되고 구매자 목록에 노출된다.

**기본 흐름**
1. 판매자가 상품 정보(이름, 설명, 가격, 재고 수량, 이미지)를 입력한다.
2. 시스템이 필수값·형식(가격/재고는 0 이상 등)을 검증한다.
3. 시스템이 상품을 판매자 소유로 저장하고 "판매중"으로 설정한다.

**대안/예외 흐름**
- 2a. 필수값 누락/형식 오류 → 항목별 오류를 표시하고 재입력을 요청한다.
- 1a. 비로그인/비판매자 접근 → 로그인 유도 또는 권한 없음 처리(FR-005).

## UC-06 상품 정보 수정

- **주요 액터**: 판매자
- **목적**: 자기 상품의 가격·재고 등을 수정한다.
- **관련 요구사항**: FR-007, FR-009, FR-023(기존 주문 스냅샷 불변)
- **사전조건**: 판매자가 해당 상품의 소유자.
- **사후조건(성공)**: 변경된 정보가 즉시 반영된다. 이미 결제된 주문에는 영향 없음.

**기본 흐름**
1. 판매자가 자기 상품을 선택해 수정 화면을 연다.
2. 판매자가 가격·재고·정보를 변경한다.
3. 시스템이 검증 후 변경을 저장하고 반영한다.

**대안/예외 흐름**
- 1a. 타 판매자 상품 접근 시도 → 권한 없음으로 거절(FR-009).
- 3a. 재고를 낮춰도 이미 확정된 주문에는 영향이 없다(FR-023).

---

## UC-07 판매 상태 전환

- **주요 액터**: 판매자
- **목적**: 상품을 판매중/판매중지로 전환한다.
- **관련 요구사항**: FR-008, FR-009
- **사전조건**: 판매자가 해당 상품의 소유자.
- **사후조건(성공)**: 판매중지 상품은 구매자 목록·검색에서 사라진다.

**기본 흐름**
1. 판매자가 상품의 판매 상태를 전환한다.
2. 시스템이 상태를 갱신한다.
3. 판매중지 시 구매자 노출 대상에서 제외한다.

**대안/예외 흐름**
- 1a. 타 판매자 상품 접근 → 권한 없음으로 거절.

## UC-08 장바구니 담기·수정

- **주요 액터**: 구매자
- **목적**: 구매할 상품과 수량을 모은다.
- **관련 요구사항**: FR-014
- **사전조건**: 구매자로 로그인.
- **사후조건(성공)**: 장바구니에 선택 상품·수량이 반영된다.

**기본 흐름**
1. 구매자가 상품 상세에서 수량을 정해 장바구니에 담는다.
2. 시스템이 장바구니에 항목을 추가/누적한다.
3. 구매자가 장바구니에서 수량을 변경하거나 항목을 삭제한다.

**대안/예외 흐름**
- 1a. 비로그인 → 로그인 유도(FR-005).
- 2a. 요청 수량이 현재 재고를 초과 → 담기 시점에는 안내만 하고, 실제 확정은 결제 시점의
  재고 선점에서 판정한다(UC-09).

## UC-09 결제 (체크아웃)  ⭐ 핵심

- **주요 액터**: 구매자
- **부가 액터**: 모의 결제 게이트웨이(외부)
- **목적**: 장바구니 상품을 결제해 주문을 확정한다.
- **관련 요구사항**: FR-015, FR-017, FR-018, FR-019, FR-020, FR-021, FR-022, FR-023, FR-024
- **관련 성공 기준**: SC-001, SC-003, SC-004, SC-005
- **사전조건**: 구매자로 로그인, 장바구니에 1개 이상 항목.
- **사후조건(성공)**:
  - 주문이 "결제완료" 상태로 생성된다.
  - 결제 수량만큼 재고가 확정 차감된다.
  - 여러 판매자 상품이면 판매자별 하위주문으로 분리된다.
  - 결제 시점 가격·상품 정보가 스냅샷으로 보존된다.
- **사후조건(실패)**: 주문이 생성되지 않고 선점 재고는 해제(또는 미차감)되어 재고가 보존된다.

**기본 흐름 (해피패스)**
1. 구매자가 배송지 정보를 입력하고 결제를 요청한다(멱등키 동반).
2. 시스템이 결제 금액을 서버에서 계산·검증한다(클라이언트 금액 불신, FR-019).
3. 시스템이 주문 수량만큼 재고를 선점한다(TTL 10분, FR-017).
4. 시스템이 멱등키로 결제를 처리한다 — 모의 결제 게이트웨이에 승인 요청(FR-018, FR-024).
5. 결제 성공 시, 시스템이 선점을 확정 차감으로 전환하고 주문을 "결제완료"로 생성한다.
6. 시스템이 판매자별 하위주문으로 분리하고 상품 스냅샷을 저장한다(FR-021, FR-023).
7. 시스템이 결제 완료와 주문 번호를 구매자에게 안내한다.

**대안/예외 흐름**
- 3a. **재고 부족/소진**(동시 구매 경합 포함): 선점 실패 → 결제를 거절하고 품절을
  안내한다. 재고는 변하지 않는다(FR-020, SC-003).
- 4a. **결제 실패**(게이트웨이 거절): 선점을 해제하고 주문을 생성하지 않는다. 실패 사유를
  안내한다.
- 4b. **중복 요청/재시도**(동일 멱등키): 이미 처리된 결제면 기존 결과를 그대로 반환하고,
  결제·주문을 다시 만들지 않는다(FR-018, SC-004).
- 1a. **결제 미완료 이탈**: 선점만 되고 결제가 끝나지 않으면 10분(TTL) 후 선점이 자동
  해제되어 재고가 복원된다(FR-017, SC-005).
- 2a. 배송지 정보 누락/형식 오류 → 항목별 오류를 안내하고 재입력을 요청한다.

**비즈니스 규칙**
- 결제 금액은 항상 서버가 산출한 값을 기준으로 한다.
- 하나의 주문 = 하나의 멱등 결제. 재시도가 겹쳐도 효과는 정확히 한 번.
- 초과 판매는 절대 허용하지 않는다.

## UC-10 내 주문 조회 (구매자)

- **주요 액터**: 구매자
- **목적**: 자신의 주문과 상태를 확인한다.
- **관련 요구사항**: FR-016, FR-022
- **관련 성공 기준**: SC-006
- **사전조건**: 구매자로 로그인.
- **사후조건(성공)**: 자신의 주문 목록·상태가 최신으로 표시된다.

**기본 흐름**
1. 구매자가 내 주문 목록을 연다.
2. 시스템이 구매자의 주문을 상태·금액·판매자와 함께 보여준다.
3. 구매자가 주문 상세를 열면 판매자별 하위주문과 각 상태를 구분해 보여준다.

**대안/예외 흐름**
- 2a. 주문 이력 없음 → 빈 목록 안내.

## UC-11 내 주문(하위주문) 조회 (판매자)

- **주요 액터**: 판매자
- **목적**: 자신에게 배정된 하위주문을 확인한다.
- **관련 요구사항**: FR-009, FR-010, FR-021
- **관련 성공 기준**: SC-007
- **사전조건**: 판매자로 로그인.
- **사후조건(성공)**: 판매자 자신에게 해당하는 하위주문만 표시된다.

**기본 흐름**
1. 판매자가 자기 주문 목록을 연다.
2. 시스템이 그 판매자에게 배정된 하위주문만 보여준다.

**대안/예외 흐름**
- 2a. 타 판매자 주문 접근 시도 → 노출·접근 차단(FR-009).

## UC-12 배송 상태 전이

- **주요 액터**: 판매자
- **목적**: 하위주문을 "배송중" → "배송완료"로 갱신한다.
- **관련 요구사항**: FR-011, FR-022
- **관련 성공 기준**: SC-006
- **사전조건**: 판매자가 해당 하위주문의 소유자, 주문이 "결제완료" 이상 상태.
- **사후조건(성공)**: 상태가 전이되고 전이 시각·이력이 감사 로그로 기록된다. 배송완료 시
  완료 시각이 남아 향후 정산의 트리거가 된다.

**기본 흐름**
1. 판매자가 하위주문의 상태를 "배송중"으로 바꾼다.
2. 시스템이 상태를 갱신하고 구매자 주문 화면에도 반영한다.
3. 판매자가 이후 "배송완료"로 바꾼다.
4. 시스템이 배송완료 시각을 기록하고 상태 전이를 감사 로그에 남긴다.

**대안/예외 흐름**
- 1a. 허용되지 않은 상태 전이(예: 결제완료 → 배송완료로 건너뛰기) → 거절하고 유효한
  다음 상태만 허용한다.
- 1b. 타 판매자 하위주문 접근 → 권한 없음으로 거절(FR-009).
