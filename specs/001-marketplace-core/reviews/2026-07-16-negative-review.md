**작성일**: 2026-07-16 · **관련 이슈**: #2 · **대상**: 001-marketplace-core 전체 설계 + 추적 매트릭스

**방법**: 4개 페르소나(설계자·개발자·구매자·판매자)를 서브에이전트로 **병렬·네거티브(결함

traceability-matrix/contracts/erd.dbml/schema.sql/diagrams/*, ADR 11건, 헌장.

## 1. 총평

설계 서사·다이어그램·추적 매트릭스는 완결적으로 보이나, ① 가장 비싼 아키텍처 결정(경계·전달
계층·원자성)이 근거 없이 낙관적으로 통과됐고, ② 커머스의 절반(결제 이후 = 취소·환불·정산)이
통째로 비어 있으며, ③ 현재 tasks.md로는 US1(핵심 결제)을 착수조차 못 하는 공백이 있다.

| 코드 | 결함 | 지적 페르소나 | 요지 |
|---|---|---|---|

| **D** | 감사 원장 테이블 부재(원칙 IV 위반) | 설계자·개발자 | "주문에 무슨 일이 있었는지 재구성" 요구하나 audit/event 테이블 없음. SLF4J 로그뿐 |

## 3. 심각도별 통합 결함 목록

### High
1. **재고 선점 원자성 ↔ 컨텍스트 분리 배타** — ReservationService가 products.reserved를 직접
   UPDATE(컨텍스트 침범, ADR-0010 위반). 단일 DB에서만 원자적. (설계자#1, 개발자#9)
2. **조건부 UPDATE ↔ JPA @Version·더티체킹 충돌** — 네이티브 UPDATE가 version 미증가·엔티티
   stale로 초과판매 재현 가능. version "보조" 정의 모호. (개발자#1)
3. **체크아웃 사가 전환 리스크 과소평가** — 4개 컨텍스트 단일 @Transactional, 부분 실패·보상
   미정의. "선점·멱등으로 대비"는 재시도 안전성일 뿐 분리 원자성 아님. (설계자#2)
4. **Data REST 자동 노출로 불변식·권한 우회** — 위 C. 프로젝션 미설계, 소유권 인가 난이도.
   ProductOpsController vs ProductService 책임 경계 모호(T031 vs T032). (설계자#3, 개발자#6)
5. **감사 원장 테이블 부재** — 위 D. order_event/state_transition_log 없음. (설계자#4, 개발자#2)
6. **payments.order_id NOT NULL ↔ 승인 후 주문 생성 모순** — REJECTED 결제를 기록할 수 없어
   실패 결제 감사 불가. (설계자#5)
7. **멱등키 리로드 이중주문** — /checkout 로드마다 새 키 발급 → 두 탭 제출 시 서로 다른 키 →
   주문 2건. SC-004가 리로드에서 깨짐. (설계자#6)
8. **멱등키 경합 창(race window)** — 동시 2요청 시 첫 요청 미커밋(order_id=NULL)을 두 번째가
   읽으면 반환할 결과 없음. STARTED/DONE 상태·대기 정책 미정의. (개발자#3)
9. **프론트 MVC ↔ 백엔드 REST 인증 전파·트랜잭션 경계 전무** — 소유권 식별 전파 방법 없음.
   체크아웃 오케스트레이션 주체(백엔드 1회 vs MVC 다회 호출) 불명 → 원자성 붕괴 위험. (개발자#4)
10. **백엔드 REST URL 계약 부재** — contracts/http-routes.md는 SSR 경로만. Data REST /products
    ↔ SSR /products 충돌, base-path 격리 없음 → 프론트 착수 불가. (개발자#5)
11. **결제 이후 도메인 부재** — 위 A. 취소·환불·정산·수수료율·정산계좌·claw-back 자리 없음.
    (고객#1, 판매자#1-3,#6)
12. **알림·배송추적 부재** — 주문확인/배송 알림 0, 송장·택배사·추적 0. 배송완료를 판매자
    일방 토글(구매자 반박 수단 없음). (고객#4,#5, 판매자#5)

### Med
13. **지연 판정 ↔ reserved 역정규화 모순** — 둘 양립 불가, 스윕 지연 창 실제로 안 닫힘(SC-005).
    (설계자#7, 개발자#7)
14. **@Scheduled 스윕 다중 인스턴스 이중 감소** — 분산 락/리더 선출 없음. 결제 성공 직전 스윕이
    HELD를 RELEASED하면 초과판매. (개발자#7)
15. **Delivery 1:1 SubOrder ↔ ADR-0008 부분배송 근거 자기모순** — 부분배송은 1:N인데 1:1로 못박음.
    (설계자#8)
16. **OrderStatus 집계 역방향 쓰기 + 가드 중첩** — delivery→order 갱신(경계 위반), PAID→SHIPPED
    가드가 COMPLETED와 중첩(비결정). (설계자#9)
17. **판매자 stock 하향 시 reserved 불변식 붕괴 / SUSPENDED 판매자 상품 계속 구매** — 가드 없음.
    (설계자#10)
18. **동시성/TTL 테스트를 단위에 배치 + Clock 주입 부재** — DB 없이 재현 불가, 시간 조작 불가.
    (개발자#8)
19. **DB CHECK 제약 전무** — 초과판매 방지가 앱 조건부 UPDATE 한 겹에만 의존. (개발자#10)
20. **판매자명 스냅샷 누락 + 선점~결제 가격변동 UX 공백** — storeName 실시간 참조로 과거 주문
    표시 변동, 본 금액과 다른 청구 가능. (개발자#11)
21. **배송비 필드 자체가 데이터모델에 없음** — 멀티셀러 배송비/무료배송/도착예정 전무. (고객#3)
22. **비회원 구매 불가 + 비번 재설정·이메일 인증 없음** — 계정 상실 시 주문내역 영구 상실. (고객#6)
23. **검색이 이름만(카테고리·정렬·필터 없음) + 리뷰·평점 없음** — 신뢰·발견성 저하. (고객#7, 판매자#8)
24. **판매자 대시보드·대량등록·재고알림·정산명세 없음** — 판매자 운영 도구 공백. (판매자#4,#8)

### 운영/보안 리스크(추가)
- Data REST 공격면(기본 전량 노출), BFF↔백엔드 CSRF/세션 모델 미정(원칙 II ✅ 처리 부적절).
- 게이트웨이 timeout/unknown 결과 상태가 어댑터 계약(payment-gateway.md)에 없음 → 실 PSP 전환 시
  이중결제 위험.

## 4. 문서 불일치 / 추적성 오류 (저비용 정정 대상)

  본문·data-model·ADR-0008은 "SubOrder 무상태, Delivery가 담당". 요약표가 정반대.
- **ADR-0004 stale**: 동시성 결정을 research R1로 확정했으나 "후속 ADR" 승격 안 됨.

  모놀리스"·Complexity Tracking "위반 없음"으로 자평 → 거버넌스 위반.
- **UC-03/04 스토리 매핑**: usecases는 US1, traceability·tasks는 US2(Phase 4)에 배치. 구매자
  탐색이 판매자 스토리 단계에서 구현되도록 꼬임.

  - SC-001/002(3분/5분) 대응 테스트는 시간 미측정 → "SC 전부 검증"은 사실과 다름.
  - §6 "갭 점검: 없음"은 거짓 → 매트릭스를 SSOT로 신뢰 불가.

## 5. 페르소나별 한 줄 총평

- **설계자**: "간판은 추출 가능 모듈러 모놀리스지만 세 코어(초과판매·체크아웃·상태집계)가 단일
  DB 원자성에 묶여 분리 순간 재작성 강제 + Data REST/감사원장 부재가 무결성·추적성 방어선을
  사후 태스크로 미룬 설계."
- **개발자**: "재고 선점(조건부 UPDATE vs @Version)과 멱등 경합 창만으로 초과판매·이중결제가
  실제로 뚫리고, 감사 로그 테이블·백엔드 REST 계약이 빠져 현재 tasks.md로는 US1을 착수 못 함."
- **구매자**: "결제까지는 3분인데, 잘못 사면 취소도 못 하고 문의할 곳도 배송 추적도 없다 —
  파는 쪽 편의만 있고 산 뒤의 나는 없는 쇼핑몰."
- **판매자**: "물건은 팔리는데 돈은 언제 어떻게 받는지 설계에 존재하지 않는다 — 배송완료를
  손으로 눌러도 정산 레코드 하나 안 생긴다."

## 6. 권고 후속 (계층별, 미적용)

- **계층 1 — 문서 정정(저비용·고가치)**: §4의 불일치·추적성 오류 수정.
- **계층 2 — 설계 보강(구현 전 필수)**: payments.order_id nullable+실패 결제 기록, 멱등 상태기계,
  감사 원장 테이블, 백엔드 REST 계약+/api base-path, Data REST read-only+커스텀 쓰기, 재고 단일
  전략 확정(@Version 제거)+CHECK 제약+Clock 주입, 동시성 테스트 통합 재분류, 스윕 분산 락.
- **계층 3 — 범위 재검토(제품 결정)**: 최소 커머스 하한선(배송 전 취소·주문확인 알림·배송추적)
  과 정산 최소 골격(Settlement+수수료+정산계좌)을 이번 범위로 당길지.

## 7. 기록된 결정 (이번 리뷰 기준)

- **아키텍처: 모듈러 모놀리스로 확정.** "컨텍스트별 별도 인스턴스 분리"는 **Non-goal**로 강등한다.
  단일 배포·단일 DB 원자성·컨텍스트 간 조인 허용(단, 상호작용은 서비스 인터페이스 경유)을
  전제로 결함 B(모순)·C·D의 상당 부분 압박이 해소된다.
  - 후속: [ADR-0010](../../../docs/adr/0010-extraction-ready-modular-monolith.md)을 "모듈러
    모놀리스 확정, 인스턴스 분리 Non-goal"로 개정하고, ADR-0011의 REST/BFF 근거를 이 전제에
    맞춰 재평가하는 것을 권고(별도 작업).
- **조치 범위: 이번에는 본 보고서 저장까지.** §4·§6의 설계 수정은 사용자 결정에 따라 보류.
