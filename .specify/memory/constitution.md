# ShopFlow 헌장 (Constitution)

## 핵심 원칙 (Core Principles)

### I. 테스트 우선 품질 (Test-First Quality, 타협 불가)

모든 비즈니스 로직에 TDD를 필수로 적용한다. 테스트를 먼저 작성하고 검토한 뒤, 실패하는 것을
확인하고(Red) 통과하도록 구현한 다음(Green) 리팩터링한다. 커머스 핵심 흐름 — 장바구니,
결제(checkout), 주문 생성, 재고 차감, 환불 — 은 병합 전에 반드시 자동화된 테스트를 갖춘다.
순수 도메인 로직(가격, 할인, 세금, 재고)은 웹 계층·영속 계층과 독립적으로 단위 테스트가
가능해야 한다. 통과하는 테스트를 함께 커밋하지 않은 기능은 "완료"가 아니다.

근거: 이커머스 흐름의 회귀 결함은 곧바로 돈과 신뢰의 손실로 이어진다. 테스트는 조용히
발생하는 주문·재고 훼손을 막는 유일하고 지속적인 방어선이다.

### II. 결제·보안 신뢰성 (Payment & Security Reliability, 타협 불가)

결제 정확성과 데이터 보호는 편의성·개발 속도보다 우선한다. 규칙:
- 결제 금액은 반드시 서버에서 계산·검증한다. 클라이언트가 보낸 가격·합계는 절대 신뢰하지
  않는다.
- 원본 카드 정보(PAN/CVV)를 저장하지 않는다. 카드 처리는 규정을 준수하는 PSP(결제대행사)에
  위임하며 PCI-DSS 적용 범위를 최소화한다.
- 시크릿(PSP 키, DB 자격증명)은 환경변수/시크릿 설정에서 주입하며 소스에 커밋하지 않는다.
- 인증, 인가, CSRF 방어, 입력 검증은 상태를 변경하는 모든 엔드포인트에서 강제한다. 개인정보는
  전송 구간과 저장소 모두에서 보호한다.

근거: 결제·데이터 사고 한 번은 쇼핑몰에 치명적이다. 이는 나중에 다룰 강화 과제가 아니라
타협 불가한 기본선이다.

### III. 데이터 무결성·트랜잭션 일관성 (Data Integrity & Transactional Consistency)

돈·재고·주문에 영향을 주는 작업은 반드시 트랜잭션으로 원자적으로 처리한다. 재고 차감, 주문
생성, 결제 승인(capture)은 부분 상태를 남겨서는 안 되며, 실패 시 깔끔하게 롤백해야 한다.
한정 수량에 대한 동시 구매는 명시적 동시성 제어(낙관적/비관적 잠금)로 처리한다 —
초과 판매(overselling)는 결함이다. 금액 값은 반드시 고정 소수점 타입을 쓴다(이진 부동소수점
금지).

근거: 커머스 데이터는 실제 돈과 실제 재고의 원장(system of record)이다. 이곳의 불일치는
겉만 잘못된 버그가 아니라 되돌릴 수 없는 손상이다.

### IV. 관측 가능성·감사 추적성 (Observability & Auditability)

요청·서비스·영속 계층 전반에 구조화된 로깅을 필수로 둔다. 모든 주문 생명주기 전이와 결제
이벤트는 상관관계(correlation)/추적(trace) id를 포함한 감사 기록을 남겨, 디버거 없이도
"이 주문에 무슨 일이 있었는지" 재구성할 수 있어야 한다. 오류는 조치 가능한 맥락과 함께
기록하고, 민감 정보는 로그에서 마스킹한다.

근거: 고객 지원, 부정거래 조사, 장애 대응은 사후에 어떤 주문·결제든 추적할 수 있어야
가능하다.

### V. 단순성·계층형 아키텍처 (Simplicity & Layered Architecture)

단순하게 시작하고 YAGNI를 지킨다. 애플리케이션은 명확한 계층 분리를 유지한다(컨트롤러/표현
→ 서비스/도메인 → 리포지토리/영속). 표현(템플릿) 계층에는 비즈니스 로직을 두지 않는다. 새
프레임워크·캐시·추상화는 구체적이고 입증된 필요가 있을 때만 도입하며, 그 근거를 기록한다.

근거: 이해하기 쉬운 계층형 코드베이스라야 쇼핑몰이 커져도 위의 테스트 우선·보안 보장을
계속 강제할 수 있다.

### VI. 한글·직관적 표현 산출물 (Korean-Language & Plain-Expression Deliverables)

모든 산출물은 한글로 작성한다. 사양서, 사용자 스토리, 계획, 태스크, 리뷰, 커밋 메시지, UI
문구, 오류 메시지, 사용자 대상 문서를 포함한다. 표현은 복잡한 용어나 장황한 문장 대신 누구나
직관적으로 이해할 수 있게 쉽고 간결하게 쓴다. 규칙:
- 산출물의 서술·설명은 한글을 기본으로 한다.
- 코드 식별자, 명령어, 라이브러리·프레임워크 고유명, 확립된 기술 약어(예: TDD, PSP,
  HTTP)는 원문 그대로 둔다 — 억지 번역으로 의미를 흐리지 않는다.
- 필요한 곳에는 짧은 한글 설명을 덧붙여 전문 용어의 의미를 직관적으로 풀어 준다.
- 한 문장은 한 가지 뜻만 담고, 불필요한 수식·중복 표현을 피한다.

근거: 팀과 이해관계자가 산출물을 오해 없이 빠르게 이해해야 협업과 검수가 빨라지고, 쉬운
표현은 결함과 소통 비용을 줄인다.

### VII. 마켓플레이스 정산·주문 생명주기 무결성 (Marketplace Settlement & Order-Lifecycle Integrity)

여러 판매자(입점사)가 참여하는 마켓플레이스로서, 돈과 이행(fulfillment)에 관한 상태는
판매자 단위로 관리하고 명시적 생명주기로 다룬다. 규칙:
- 판매자별 범위: 배송, 정산, 환불은 판매자(seller/입점사) 단위로 관리·집계한다. 하나의
  주문이 여러 판매자 상품을 포함하면 판매자별 하위 주문(이행 단위)으로 분리해 각각의 배송·
  정산·환불을 독립적으로 추적한다.
- 재고 선점: 결제 시작 전에 재고를 선점(reserve)하고 만료 시간(TTL)을 둔다. 결제가
  완료되지 않으면 선점을 자동 해제하고, 결제 성공 시 선점을 확정 차감으로 전환한다. 이로써
  초과 판매를 막고(원칙 III 연계) 미결제 재고 묶임을 방지한다.
- 멱등 결제: 결제·재고 확정·정산·환불처럼 돈이나 상태를 바꾸는 요청은 멱등키(idempotency
  key)로 처리한다. 재시도가 겹치거나 중복 요청이 와도 효과는 정확히 한 번만 발생하며
  (exactly-once), 중복 요청에는 최초와 동일한 결과를 반환한다.
- 정산·수수료 공제: 배송 완료 등 확정 트리거가 발생하면 판매자 정산 금액을 산출하고
  플랫폼 수수료를 공제한다. 정산·수수료·지급은 감사 가능한 원장(ledger) 기록으로 남긴다
  (원칙 IV 연계).
- 환불·회수: 환불은 원 결제 및 정산과 연결해 부분/전체를 추적한다. 이미 정산이 이뤄진
  건을 환불하면 판매자 정산에서 회수(claw-back)를 반영한다.
- 주문 상태 전이는 명시적 상태 기계(state machine)로 정의하고, 각 전이는 감사 기록을
  남긴다.

근거: 마켓플레이스에서는 "누구의 상품을, 얼마에, 언제, 어떻게 이행하고 정산했는가"가
곧 신뢰이자 회계다. 판매자별 분리·재고 선점·멱등 처리·정산 원장은 이중 결제·초과 판매·
정산 오류 같은 되돌리기 어려운 사고를 구조적으로 예방한다. (요구사항 확장 시 이 원칙의
범위 안에서 정합성을 유지한다.)

## 기술·보안 제약 (Technology & Security Constraints)

- 백엔드: Java + Spring Boot (Spring MVC, Spring Data, Spring Security).
- 표현(프레젠테이션): 서버 사이드 템플릿 엔진(예: Thymeleaf)으로 HTML 렌더링. 원칙 V에 따라
  뷰 템플릿은 로직을 최소화한다.
- 영속(persistence): 리포지토리 계층을 통해 관계형 데이터베이스에 접근한다. 스키마 변경은
  버전 관리되는 마이그레이션을 거친다.
- 결제: 규정을 준수하는 외부 PSP(결제대행사)를 연동한다. 자체 카드 정보 저장은 하지 않는다
  (원칙 II).
- 설정: 환경별 값(포트, DB, PSP 키)은 환경변수 / Spring 프로파일로 외부화하고 기본값(fallback)을
  문서화한다. 포트·시크릿을 하드코딩하지 않는다.
- 상태를 변경하는 모든 엔드포인트는 인증, 인가, CSRF 방어, 서버측 검증을 강제한다.

## 개발 워크플로·품질 게이트 (Development Workflow & Quality Gates)

- 모든 변경은 리뷰를 거쳐 반영한다. 리뷰어는 다섯(여섯) 원칙 준수 여부를 확인하며, 특히
  테스트 우선(I)과 결제·보안(II)을 중점 검토한다.
- 병합 게이트: 변경된 로직에 대한 테스트 작성·통과, 보안 민감 변경(인증, 결제, 가격, 재고)의
  명시적 표기·리뷰, diff 내 시크릿 없음.
- 원칙에서 벗어나는 부분은 반드시 PR/계획의 "복잡성 / 예외(Complexity / Deviation)" 노트에
  근거를 적고 승인받는다. 그렇지 않으면 변경을 반려한다.
- 계획 템플릿의 Constitution Check는 각 기능 계획을 설계 전과 구현 전에 게이트로 검증한다.

## 거버넌스 (Governance)

이 헌장은 ShopFlow의 다른 개발 관행보다 우선한다. 개정은 반드시 문서로 제안·검토·승인하며,
버전 상향을 포함하고, 동작이 바뀌는 경우 의존 템플릿 전반에 대한 마이그레이션/전파 노트를
포함한다.

버전 정책(시맨틱 버저닝):
- MAJOR: 하위 호환이 깨지는 거버넌스 변경 또는 원칙 제거·재정의.
- MINOR: 새 원칙/섹션 추가 또는 지침의 실질적 확장.
- PATCH: 표현 정리, 오탈자 수정 등 의미를 바꾸지 않는 다듬기.

준수: 모든 PR과 리뷰는 이 원칙들의 준수를 확인해야 하며, 정당화되지 않은 복잡성은 반려한다.
이 파일이 권위 있는 원천(single source of truth)이다. 에이전트/런타임 가이드 문서(예:
CLAUDE.md)는 이 헌장과 일관되게 유지한다.
